# GRPO Training with NVIDIA NeMo RL on Amazon SageMaker

Train [Qwen2.5-1.5B](https://huggingface.co/Qwen/Qwen2.5-1.5B) on math reasoning using GRPO (Group Relative Policy Optimization) with [NVIDIA NeMo RL](https://github.com/NVIDIA/NeMo-RL) v0.5.0 on Amazon SageMaker Training Jobs.

## Overview

GRPO is a reinforcement learning algorithm for post-training LLMs:

1. **Generate** multiple responses per prompt using vLLM
2. **Evaluate** responses with a reward function (math correctness via `math_verify`)
3. **Update** the policy using group-relative advantages

This example uses Ray for distributed orchestration across GPUs, with colocated generation and training on the same nodes.

| Component          | Details                          |
| ------------------ | -------------------------------- |
| Base model         | Qwen/Qwen2.5-1.5B                |
| Dataset            | OpenMathInstruct-2 (HuggingFace) |
| Reward             | `hf_math_verify`                 |
| Generation backend | vLLM                             |
| Policy training    | DTensor (FSDP v2)                |
| Instance type      | ml.p5.48xlarge (8x H100 80GB)    |
| NeMo RL version    | v0.5.0                           |

## Prerequisites

- AWS account with access to `ml.p5.48xlarge` instances
- Docker installed locally (or SageMaker Studio with Docker access)
- AWS CLI configured with appropriate permissions
- Python 3.10+ with dependencies:
  ```
  pip install -r scripts/requirements.txt
  ```

## Quick Start

### 1. Build and push the Docker image

The custom container extends the NGC NeMo RL image with EFA drivers, AWS OFI NCCL plugin, and SageMaker training toolkit.

```bash
cd docker
./create-image.sh nemo-rl-sagemaker
```

This will:

- Build the Docker image from the NGC base (`nvcr.io/nvidia/nemo-rl:v0.5.0`)
- Create an ECR repository if it doesn't exist
- Push the image to ECR

The build takes approximately 10-15 minutes.

### 2. Launch the training job

Open and run the notebook:

```
1-grpo-training.ipynb
```

The notebook will:

1. Generate the GRPO configuration (`grpo.yaml`) and upload it to S3
2. Configure the SageMaker `ModelTrainer` with the custom container
3. Launch the training job

### 3. Monitor training

**CloudWatch Logs**: Available in the SageMaker console under the training job.

**Ray Dashboard** (optional): Enable remote debug in the notebook and use SSM port forwarding:

```bash
aws ssm start-session \
  --target sagemaker-training-job:<job-name>_algo-1 \
  --document-name AWS-StartPortForwardingSession \
  --parameters '{"portNumber":["8265"],"localPortNumber":["8265"]}'
```

Then open `http://localhost:8265/` in your browser.

**TensorBoard**: Logs are written to the S3 output path under `data/tb_logs/`.

## How It Works

### Docker Container

The Dockerfile extends the NGC NeMo RL v0.5.0 image:

1. Installs **EFA drivers** and **AWS OFI NCCL** plugin for high-performance inter-node communication
2. Syncs all NeMo RL extras (vLLM, NeMo AutoModel, flash-attn, etc.) into the main Python venv using `uv sync`
3. Bundles **Prometheus** for Ray metrics collection
4. Installs the **SageMaker training toolkit**

The environment variable `NEMO_RL_PY_EXECUTABLES_SYSTEM=1` forces all Ray actors to use the main venv instead of the separate uv-managed venvs in the NGC image.

### Ray on SageMaker

The `launcher.py` script (from [sample-ray-on-amazon-sagemaker-training-jobs](https://github.com/aws-samples/sample-ray-on-amazon-sagemaker-training-jobs)) bootstraps a Ray cluster on SageMaker:

1. Starts `ray start --head` on the first node
2. Workers join via `ray start --address=<head>:<port>`
3. Loads the training script via `--entrypoint train_grpo.py`

NeMo RL connects to the cluster with `ray.init(address="auto")`.

### Training Script

`train_grpo.py` loads the GRPO config, sets up the dataset and environment, and calls `grpo_train()` from NeMo RL. Key components:

- **Data**: `OpenMathInstruct-2` loaded via HuggingFace datasets
- **Reward**: Math answer verification using `hf_math_verify`
- **Policy**: DTensor-based training with sequence packing
- **Generation**: vLLM with colocated generation (same GPUs as training)

## Configuration

The `grpo.yaml` configuration is generated by the notebook. Key parameters to adjust:

| Parameter                          | Default           | Description                         |
| ---------------------------------- | ----------------- | ----------------------------------- |
| `grpo.max_num_steps`               | 100               | Total training steps                |
| `grpo.num_prompts_per_step`        | 32                | Prompts per training step           |
| `grpo.num_generations_per_prompt`  | 16                | Responses generated per prompt      |
| `policy.model_name`                | Qwen/Qwen2.5-1.5B | HuggingFace model ID                |
| `policy.max_total_sequence_length` | 512               | Max sequence length                 |
| `policy.precision`                 | bfloat16          | Training precision                  |
| `cluster.gpus_per_node`            | 8                 | Must match instance GPU count       |
| `cluster.num_nodes`                | 1                 | Must match SageMaker instance count |

## Multi-Node Training

To scale to multiple nodes, update in the notebook:

1. Set `instance_count` to the desired number of nodes
2. Update `cluster.num_nodes` in the GRPO config to match
3. Ensure `NCCL_IB_DISABLE=0` for InfiniBand/EFA networking

## Checkpointing

Checkpoints are saved to `/opt/ml/checkpoints/` inside the container, which maps to the S3 checkpoint path configured in the `ModelTrainer`. The configuration keeps the top 3 checkpoints by validation accuracy.
